# Fork / 自定义与优化（企业规则不同怎么办）

当企业的检查规则、交付物命名习惯、阶段划分与本仓库默认假设不一致时，推荐通过“规则资产化 + 小步迭代调优”的方式自定义。

## 建议做法（不改代码也能先落地）

1. Fork 本仓库到你们自己的组织/项目仓库
2. 先只维护两份 Excel：
   - `rules.xlsx`：定义检查项（规则）本身
   - `config.xlsx`：定义各项目/系统启用哪些检查项
3. 在 UI 中用“下载模板/下载示例”生成基线结构，再逐步替换为企业自己的字段与检查清单

这种方式的优势是：不需要开发即可上线试运行，且规则/配置可作为质量体系资产沉淀。

## 规则自定义（rules.xlsx）

优先把“企业差异”体现在规则上：

- 分阶段：用 Sheet 名表达阶段（例如：立项/需求/设计/测试/上线/运维），每行一条规则
- 限定后缀：给规则加上后缀（如 `.docx,.xlsx`），减少误匹配
- 区分检查对象：
  - `内容`：需要读正文并调用大模型做语义检查
  - `文件名`：只检查交付物是否存在/命名是否合理（更稳、成本更低）
- 规则要点写法（更利于自动化）：
  - 明确交付物名：在要点中写清《xxx》或“必须提供/需提交/交付 xxx”
  - 明确结构要求：例如“必须包含：范围/结论/风险/回退方案”等
  - 避免多目标混写：一个规则尽量只检查一个交付物或一个主题

## 配置自定义（config.xlsx）

企业通常存在“同一套规则在不同项目上只执行一部分”的需求：

- Sheet=阶段
- 行=项目/系统（项目编号/系统编号必须与导入目录中识别到的一致）
- 列=检查项名称（建议从 rules.xlsx 直接复制粘贴，避免不一致）
- 单元格=是否启用（√/1/true/是/需要 等都可）

未配置的项目会回退使用全部规则；已配置但某阶段未启用任何检查项时，该阶段规则会被跳过。

## 进一步优化（需要改代码的场景）

当你们的命名习惯、文件类型互斥关系更复杂，建议在 fork 仓库中做以下两类增强。

### 1) 强化候选文件匹配（减少误匹配）

候选文件匹配的“唯一口径”在后端接口 `/api/scan/candidates`，对应实现：

- [autoqa_scan.py](file:///Users/stephenwang/Documents/trae_projects/auto-QA/scripts/autoqa_scan.py)

可调优点：

- 命名技能库：调整/扩充 `NAMING_SKILL`（关键词 → 推荐命名模式），用于推导更准确的目标文件集合。
- 互斥规则：补充 `forbidden_overlap(rule_text, file_text)` 中的互斥对，排除“明显不应匹配”的组合。
- 规则意图过滤：在 `allow_file_for_rule(file_text)` 中加入行业/企业特有的文件类别限制（例如某些文件只对某类规则开放）。

### 2) 调整提示词模板（提高可复核性）

系统会为每条规则生成提示词，并要求大模型返回 JSON（结论/原因/建议/证据）。

可调优点：

- `work/inputs/autoqa.skill.prompt.txt`：可作为企业定制的提示词模板入口（后端会读取该文件，存在则使用）。  
  建议要求“证据必须给出文件路径 + 摘录”，并禁止无证据结论。

## 推荐的调优闭环

1. 先用 `scope=文件名` 跑通“交付物齐套率”检查（把缺件先压下去）
2. 对误匹配多的规则：补后缀、补交付物名（《...》）、补关键词
3. 对需要内容判断的规则再改为 `scope=内容`，逐条观察证据质量
4. 发现“明显不应匹配”的情况，就把互斥规则补到后端（形成企业级知识库）
