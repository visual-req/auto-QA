# Scan（扫描过程）

本文说明扫描过程的关键机制：项目编号识别、规则与候选文件匹配（含缺少文件口径）、文件内容提取流程。

## 扫描流程概览

1. 导入目录（通常选择 `work/inputs`），系统会收集可扫描文件列表（过滤模板/隐藏文件与不支持类型）。
2. 识别每个文件所属项目与阶段（由路径/文件名推断）。
3. 按“项目 → 规则”逐条执行：
   - 先通过后端候选推导接口，为该规则挑选最可能的目标文件集合。
   - 若候选为空：直接判定“缺少文件”，不进入内容检查/大模型。
   - 若规则为“文件名/存在性”类：只确认候选存在，不检查正文。
   - 若规则为“内容”类：提取候选文件正文（必要时调用后端提取），再调用大模型判断并产出证据。

## 如何识别项目编号

前端在导入文件时会从路径与文件名中提取项目编号（示例：`W0201C`、`PD0002` 等），并据此把文件归入某个项目。

识别来源与优先级：

- 优先从 `webkitRelativePath` 的目录分段中提取（例如 `W0201C/测试阶段/xxx.docx`）。
- 其次从整个相对路径文本中提取。
- 最后从文件名中提取。

主要识别模式（不完全列举）：

- `A12345`（1 字母 + 5 数字）
- `A1234B`（1 字母 + 4 数字 + 1 字母）
- `AB1234`（2 字母 + 4 数字）
- 兜底：`[A-Za-z]{1,3}\d{4,6}[A-Za-z]?`

阶段识别规则：

- 当项目编号出现在路径中时，阶段通常取项目目录的下一层目录名（例如 `W0201C/测试阶段/...` 的“测试阶段”）。

## 如何识别规则、候选文件与“缺少文件”

### 1) 规则来源

规则来自 `rules.xlsx`（按 Sheet=阶段解析）。每条规则至少包含“检查要点/规则详细”，并可包含：

- `stage`（阶段）
- `extensions`（后缀限制）
- `scope`（检查对象：内容/文件名）
- `promptMeta`（从命名技能推断出的命名模式与关键词，用于候选文件推导）

### 2) 候选文件推导（后端唯一口径）

每条规则会调用后端接口 `/api/scan/candidates` 推导候选文件集合。核心步骤：

- 阶段过滤：优先在规则所属阶段内找文件（若该阶段无文件则回退到全项目文件）。
- 后缀过滤：规则配置了后缀时，仅保留匹配后缀的文件。
- 互斥过滤：基于规则意图与文件类型做“明显不应匹配”的排除（例如测试报告≠迁移方案、评审意见规则不拿设计说明书当对象等）。
- 命名匹配：
  - 若规则存在命名模式（`promptMeta.naming[*].namePattern`），优先按命名模式命中。
  - 若是“严格缺失模式”（例如检查要点包含《...》或包含“必须提供/交付/缺少”等信号），命名未命中会直接判定缺少文件。
- 关键词/token 匹配：从规则要点/关键词中抽取 token，与文件路径/文件名做命中计分，选择最高分文件集合。
- 降级策略：对于内容检查类规则，若无法推导到明确交付物，会降级为“同阶段/后缀文件集合”进行内容检查，避免所有规则都变成缺少文件。

### 3) 缺少文件判定

缺少文件采用“先候选、后内容”的短路策略：

- 若 `/api/scan/candidates` 返回 `candidates=[]`：直接记录“缺少目标文件”，不再提取正文、不调用大模型。
- 若大模型返回“不符合”且 reason 表达为缺少文件（例如“目标文件集合为空/未找到…文件/无法执行检查/缺少…文件”）：也归类为缺少文件。

## 文件存在性 vs 内容检查

规则的 `scope` 决定检查方式：

- `scope=file`：只判断候选文件是否存在/命名是否符合（不检查正文，也不要求证据摘录）。
- `scope=content`：需要提取正文并调用大模型做语义判断。

另外，对“签字/签署/盖章/确认完整/签字确认”等明显偏“存在性/签署齐全”的规则，会按存在性处理，不执行内容检查。

## 文件内容解析（正文提取）过程

### 1) 前端提取与缓存

内容检查类规则会对候选文件提取正文：

- 文本类文件（如 `.md/.txt/.js/.xml/...`）直接在浏览器读取文本。
- 二进制办公文件（`.docx/.pdf/.xlsx/.pptx/.xls`）通过后端接口 `/api/extract/text` 提取。

为提升性能，前端对“同一个文件”的提取结果做缓存，缓存 key 由相对路径/文件名 + size + lastModified 组成，避免重复提取。

为控制成本，单个候选文件正文在发送到大模型前会截断（默认最多 12000 字符）。

### 2) 后端提取逻辑（/api/extract/text）

后端收到文件二进制后，会落临时文件并按后缀提取：

- `.xlsx`：使用 `openpyxl` 读取所有 sheet，按行拼接为文本
- `.xls`：使用 `xlrd`
- `.docx`：优先 `python-docx`，失败时回退为解析 docx 内部 XML
- `.pptx`：优先 `python-pptx`，失败时回退为解析 slide XML
- `.pdf`：优先 `pdfplumber`，失败时尝试 `pdftotext`，再回退 `pypdf`

若无法提取到有效文本，会标记为不可读（unreadable），该文件不会进入内容检查。

## 大模型判断与输出

内容检查类规则会调用 `/api/scan/llm`，后端把“规则提示词 + 候选文件列表 + 候选正文（可能截断）”组织为对话消息，并要求模型输出一个 JSON：

```json
{"conclusion":"符合|不符合","reason":"...","suggestion":"...","evidence":[{"file":"...","snippet":"..."}]}
```

若模型输出不可解析，会回退为“无法解析 JSON”的不符合项，并保留原始输出片段用于排查。
